## Contract Alignment

This document reflects Gateway v1 canonical payload behavior. It supersedes earlier draft contracts where ID generation or update semantics differed. Specifically:

* `artifact.save` is **CREATE ONLY**
* Updates are performed **exclusively** via `artifact.update`
* `artifact_id` is generated by the database on INSERT, **never** by the Gateway or client

---

## ⚠️ CRITICAL EXECUTION RULE (Chrome Extension)

When generating payloads intended for **Chrome Extension execution**, the following rules are **non-negotiable**:

1. Output must be **raw JSON only**
2. Payload must start with `{` and end with `}`
3. **No** markdown fences (```), headings, labels, comments, or explanatory text
4. Only **one payload per execution** unless the extension explicitly supports JSON arrays
5. If a payload depends on the result of a prior save:

   * The first payload **must be executed and confirmed**
   * The returned `artifact_id` **must be extracted from the Gateway response**
   * Subsequent payloads must explicitly reference that confirmed `artifact_id`

Violation of these rules will result in execution failure or broken artifact lineage.

## 1. Purpose and Scope

### Why This Document Exists

This document defines the exact JSON payload formats required for all Gateway v1 actions to succeed on the first execution attempt. It consolidates truth from multiple authoritative sources into a single reference.

### What Problems It Prevents

* Payload guessing and trial-and-error debugging
* Field name mismatches between client and server expectations
* Incorrect optionality assumptions
* Mutation attempts on immutable or restricted fields
* Type mismatch errors from incorrect `artifact_type` handling
* Orphaned or incorrectly linked artifacts caused by assumed UUIDs

### What It Explicitly Does NOT Cover

* Workflow implementation details (n8n node configurations)
* RLS policy enforcement mechanisms
* Database trigger behavior
* Error recovery procedures
* Future actions (`artifact.delete`, `artifact.archive`)

## 2. Global Payload Rules (Applies to ALL Actions)

### 2.1 Always Required Fields

| Field             | Type    | Source | Description                                            |
| ----------------- | ------- | ------ | ------------------------------------------------------ |
| `gw_action`       | text    | Client | Action identifier (save, query, update, list, promote) |
| `gw_workspace_id` | uuid    | Client | Target workspace for the request                       |
| `artifact_type`   | text    | Client | Must be from allowed artifact types                    |
| `priority`        | integer | Client | Required for save (INSERT). Range 1–5. Default: 3     |

### 2.2 Conditionally Required Fields

| Field         | Condition                                                   | Type   | Description                |
| ------------- | ----------------------------------------------------------- | ------ | -------------------------- |
| `artifact_id` | Required for query, update, promote; **Forbidden for save** | uuid   | Target artifact identifier |
| `extension`   | Required for save, update                                   | object | Type-specific payload      |
| `title`       | Required for save on most types                             | text   | Human-readable title       |
| `transition`  | Required for promote                                        | text   | Lifecycle transition key   |

### 2.3 Forbidden from Clients (Server-Owned)

| Field                 | Reason                                     |
| --------------------- | ------------------------------------------ |
| `artifact_id` on save | Generated by database; save is CREATE ONLY |
| `gw_user_id`          | Derived from auth context                  |
| `version`             | System-managed                             |
| `created_at`          | System-managed                             |
| `updated_at`          | System-managed                             |
| `owner_user_id`       | Derived from auth context                  |
| `workspace_id`        | Derived from `gw_workspace_id`             |

## 2.4 Sequential Action Discipline (Execution Invariant)

When performing **multiple dependent actions** (e.g., project → restart, project → snapshot):

1. Execute the first action
2. Wait for Gateway confirmation
3. Extract the returned `artifact_id`
4. Use that exact ID in subsequent payloads

Clients and assistants must **never**:

* Invent UUIDs
* Assume save success
* Continue execution without confirmation

This invariant applies to **all execution surfaces**.

---

## 2.5 Priority Mandate (Effective 2026-02-16)

**`priority` (integer 1–5) is REQUIRED for all artifact INSERT operations.**

The payload layer must supply `priority` explicitly. Do NOT rely on database DEFAULT 3. If the caller does not specify a priority, the system default at the payload layer is `3`.

This requirement applies to **all artifact types**: project, journal, snapshot, restart, grass, thorn, branch, leaf, limb, instruction_pack, forest, thicket, flower.

**Database enforcement:** `priority integer DEFAULT 3 NOT NULL`, CHECK `(priority >= 1 AND priority <= 5)`.

**Rationale:** The Gateway does not rely on database defaults. All INSERT operations must be explicit at the payload layer to prevent ambiguity and ensure deterministic behavior across execution surfaces.
